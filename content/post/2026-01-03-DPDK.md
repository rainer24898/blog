# DPDK cho C/Linux: kiến trúc, pipeline xử lý gói tin và debug hiệu năng

## 1. DPDK là gì?

**DPDK (Data Plane Development Kit)** là một tập thư viện và driver trong user space để xây dựng các ứng dụng xử lý gói tin tốc độ cao trên Linux/Unix-like systems.
Nếu nhìn theo tư duy hệ thống, DPDK không chỉ là một thư viện “đọc/ghi packet nhanh”, mà là một **mô hình data plane hoàn chỉnh** gồm:

* Environment Abstraction Layer (**EAL**),
* bộ nhớ dựa trên **hugepage**,
* mô hình xử lý **poll mode**,
* driver user-space cho NIC (**PMD**),
* buffer packet (**mbuf**),
* memory pool (**mempool**),
* hàng đợi lockless/ring,
* API thiết bị mạng (**ethdev**),
* và các thư viện phụ trợ cho timer, hash, ring, flow, event, cryptography, v.v.

Nói ngắn gọn:

**DPDK là nền tảng để viết data plane hiệu năng cao trong user space, tránh overhead của kernel networking stack truyền thống.**

---

## 2. Vì sao DPDK tồn tại?

Nếu dùng Linux kernel networking stack theo cách truyền thống, gói tin thường phải đi qua nhiều lớp như:

* interrupt handling,
* softirq/NAPI,
* skb allocation,
* protocol stack,
* socket layer,
* syscall boundary,
* context switch giữa kernel và user space.

Mô hình này rất linh hoạt và phù hợp cho đa số ứng dụng chung, nhưng khi mục tiêu là:

* forwarding tốc độ rất cao,
* packet processing với latency thấp,
* NFV / UPF / vRouter / firewall / DPI / load balancer,
* hoặc các pipeline packet I/O cần sát phần cứng,

thì overhead của kernel stack trở thành nút thắt.

DPDK ra đời để giải quyết chính bài toán đó bằng cách:

* đưa packet I/O vào **user space**,
* dùng **polling** thay vì phụ thuộc nặng vào interrupt,
* truy cập queue descriptor trực tiếp qua **PMD**,
* dùng **hugepage** để tối ưu ánh xạ bộ nhớ và DMA,
* dùng **mempool/mbuf** để giảm chi phí cấp phát/giải phóng bộ nhớ,
* tối ưu **burst processing** để giảm overhead trên từng gói.

---

## 3. Tư duy đúng khi học DPDK

Một cách nhìn rất hữu ích là đừng học DPDK như danh sách API rời rạc.
Thay vào đó, nên nhìn nó như một pipeline có kiến trúc rõ ràng:

1. **EAL** khởi tạo môi trường runtime
2. ứng dụng chiếm dụng **hugepage memory**
3. NIC được bind sang **vfio-pci** hoặc PMD phù hợp
4. ứng dụng tạo **mempool** chứa các **mbuf**
5. cấu hình **ethdev port / RX queue / TX queue**
6. lcore chạy vòng lặp **poll RX burst → xử lý → TX burst**
7. packet được chuyển qua pipeline với chi phí tối thiểu

Nếu nhớ theo kiểu cực ngắn:

**EAL → hugepage → PMD → mempool/mbuf → rx burst → xử lý → tx burst**

Đó là lõi của phần lớn ứng dụng DPDK cơ bản.

---

## 4. EAL (Environment Abstraction Layer)

## 4.1 EAL là gì?

**EAL** là lớp nền của DPDK dùng để trừu tượng hóa môi trường hệ điều hành và nền tảng phần cứng.
Nó chịu trách nhiệm cho các việc như:

* khởi tạo hugepage memory,
* phát hiện CPU core và NUMA topology,
* ánh xạ bộ nhớ vật lý/IOVA,
* thiết lập multi-process support,
* parse command line options của DPDK,
* khởi tạo PCI device access,
* set up thread affinity / lcore model.

Nói ngắn gọn:

**EAL là bootstrap layer của toàn bộ ứng dụng DPDK.**

## 4.2 Vì sao EAL quan trọng?

Nếu không có EAL, ứng dụng DPDK sẽ phải tự xử lý hàng loạt chi tiết nền tảng như:

* reserve hugepages,
* map PCI BAR,
* xử lý NUMA,
* pin thread vào core,
* đồng bộ hóa môi trường runtime,
* hỗ trợ đa tiến trình.

EAL gom các việc đó lại và chuẩn hóa chúng cho ứng dụng.
Đây là lý do phần lớn chương trình DPDK đều bắt đầu bằng:

```c
int ret = rte_eal_init(argc, argv);
if (ret < 0) {
    rte_exit(EXIT_FAILURE, "EAL init failed\n");
}
```

## 4.3 EAL arguments thường gặp

Một số tham số EAL rất hay dùng trong thực tế gồm:

* `-l` hoặc `-c` để chọn lcore
* `-n` để chỉ số memory channel
* `-a` để chỉ định PCI device
* `--socket-mem` để chia hugepage theo NUMA socket
* `--huge-dir` để chỉ thư mục mount hugepage
* `--file-prefix` cho multi-process / tách namespace tài nguyên
* `--proc-type=primary|secondary`

Khi debug ứng dụng DPDK, parse sai EAL args là một lỗi rất phổ biến.

## 5. Hugepage và vì sao DPDK cần hugepage

## 5.1 Hugepage là gì?

Hugepage là các trang nhớ kích thước lớn hơn page thông thường, ví dụ 2MB hoặc 1GB trên Linux.
DPDK dùng hugepage vì các lý do quan trọng như:

* giảm áp lực lên TLB,
* giảm fragmentation,
* thuận lợi hơn cho DMA và ánh xạ memory lớn,
* giảm overhead quản lý page table,
* giúp cấp phát vùng nhớ lớn cho mempool/buffer hiệu quả hơn.

## 5.2 Vì sao DPDK thích hugepage?

Data plane tốc độ cao thường cần:

* vùng nhớ lớn,
* truy cập đều và liên tục,
* DMA mapping hiệu quả,
* locality tốt,
* latency thấp.

Hugepage là rất phù hợp cho các mục tiêu đó.
Vì vậy, hầu như mọi ứng dụng DPDK nghiêm túc đều phải quan tâm đến hugepage setup.

## 5.3 Góc nhìn debug

Khi ứng dụng DPDK không chạy được hoặc hiệu năng bất thường, các câu hỏi đầu tiên cần soi là:

* hệ thống đã reserve đủ hugepage chưa?
* mount hugetlbfs đúng chưa?
* `--socket-mem` có hợp lý không?
* process có quyền truy cập hugepage không?
* memory có đang nằm sai NUMA socket không?

---

## 6. VFIO, UIO và binding NIC

## 6.1 Vì sao phải bind NIC sang driver phù hợp?

DPDK muốn truy cập queue và thanh ghi thiết bị theo mô hình user space / PMD.
Nếu NIC vẫn bị kernel net driver quản lý như bình thường, ứng dụng DPDK sẽ không điều khiển trực tiếp được data plane path đó.

Do đó, các cổng dùng cho DPDK thường phải được bind sang driver phù hợp như:

* **vfio-pci**,
* hoặc trong một số môi trường cũ hơn là **uio_pci_generic / igb_uio**.

## 6.2 Vì sao vfio-pci quan trọng hơn trong thực tế hiện đại?

Trong các hệ thống Linux hiện đại, đặc biệt khi có **UEFI secure boot**, tài liệu DPDK khuyến nghị dùng **vfio-pci** thay vì UIO truyền thống.
Điều này liên quan đến tính an toàn, hỗ trợ IOMMU và khả năng hoạt động tốt hơn trong môi trường hệ thống hiện đại.

## 6.3 dpdk-devbind là gì?

`dpdk-devbind` là utility chính thức của DPDK để:

* xem trạng thái binding của device,
* bind device sang `vfio-pci`,
* unbind device khỏi driver hiện tại,
* chuyển thiết bị giữa kernel driver và DPDK driver context.

Ví dụ kiểu lệnh thường gặp:

```bash
sudo dpdk-devbind --status
sudo dpdk-devbind --bind=vfio-pci 0000:18:00.0
```

---

## 7. PMD (Poll Mode Driver)

## 7.1 PMD là gì?

**PMD (Poll Mode Driver)** là driver của DPDK chạy trong user space để cấu hình thiết bị và truy cập trực tiếp các RX/TX descriptor của queue mà không dựa vào interrupt cho data path thông thường.
Theo tài liệu DPDK, PMD truy cập trực tiếp RX/TX descriptors và dùng polling thay vì interrupt, ngoại trừ một số trường hợp như Link Status Change.

## 7.2 Vì sao PMD là “trái tim” của DPDK?

Nếu EAL là bootstrap layer, thì **PMD là lớp kéo packet từ NIC vào user space và đẩy packet ngược ra NIC**.
Hiệu năng thực tế của data plane phụ thuộc rất mạnh vào:

* chất lượng PMD cho NIC đó,
* vector path / scalar path,
* offload support,
* NUMA placement,
* queue design,
* burst size,
* cache locality.

## 7.3 Polling có lợi gì?

Polling loại bỏ nhiều overhead liên quan đến interrupt-driven path như:

* interrupt storms,
* context switch,
* wakeup latency,
* jitter cao.

Đổi lại, polling có chi phí:

* ăn CPU core liên tục,
* đòi hỏi pin core rõ ràng,
* cần thiết kế core loop hợp lý.

Đây là trade-off cốt lõi của DPDK.

---

## 8. ethdev là gì?

## 8.1 Vai trò của ethdev

`ethdev` là API abstraction chính của DPDK cho Ethernet device.
Thay vì viết code riêng cho từng NIC driver, ứng dụng thường thao tác thông qua các API `rte_eth_*` như:

* đếm port,
* cấu hình port,
* setup RX/TX queue,
* start/stop port,
* RX burst,
* TX burst,
* lấy statistics,
* cấu hình offload,
* kiểm tra link status.

## 8.2 Tư duy đúng

**ethdev là lớp abstraction ứng dụng nhìn thấy; PMD là lớp thực thi phía dưới.**

Đây là lý do phần lớn code packet I/O trong DPDK xoay quanh API `rte_eth_*`.

---

## 9. lcore và mô hình xử lý CPU trong DPDK

## 9.1 lcore là gì?

DPDK dùng khái niệm **lcore** để chỉ các logical core mà ứng dụng DPDK dùng cho processing.
Các lcore này thường được pin vào CPU cụ thể để:

* tránh scheduler jitter,
* giữ cache locality,
* tách rõ control core và worker core,
* tối ưu vòng lặp polling.

## 9.2 Mô hình core phổ biến

Một số mô hình quen thuộc:

* **1 core / 1 queue pair**
* **1 RX core + 1 TX core**
* **pipeline nhiều stage theo lcore**
* **RSS phân tán lưu lượng lên nhiều RX queue và nhiều worker core**

## 9.3 Vì sao pin core quan trọng?

Nếu để thread chạy tự do trên CPU mà không pin hợp lý, hiệu năng có thể suy giảm do:

* cache miss tăng,
* NUMA lệch,
* context switching,
* tăng jitter,
* mất tính dự đoán của latency.

---

## 10. NUMA và locality

## 10.1 NUMA ảnh hưởng gì tới DPDK?

Trong hệ thống nhiều socket, truy cập bộ nhớ ở socket khác có thể chậm hơn đáng kể so với local memory.
Với DPDK, NUMA ảnh hưởng trực tiếp đến:

* mempool placement,
* queue-to-core mapping,
* NIC-to-core affinity,
* RX/TX throughput,
* tail latency.

## 10.2 Nguyên tắc vàng

Một nguyên tắc rất thực dụng là:

* NIC trên socket nào,
* thì queue, mempool và worker xử lý nó nên cố gắng ở cùng socket đó.

## 10.3 Góc nhìn debug

Rất nhiều hệ thống “đúng code nhưng chậm” là do:

* PMD chạy ở socket 0,
* mempool ở socket 1,
* NIC nằm ở socket 1,
* worker lại pin sang socket 0.

Đây là kiểu lỗi cấu hình hiệu năng cực kỳ phổ biến.

---

## 11. mbuf là gì?

## 11.1 Khái niệm

`mbuf` là cấu trúc dữ liệu packet buffer trung tâm của DPDK.
Theo mô hình DPDK, packet không đi qua `sk_buff` như Linux kernel stack, mà đi qua **rte_mbuf**.

## 11.2 Vì sao mbuf quan trọng?

Mbuf mang:

* metadata của packet,
* con trỏ tới vùng data buffer,
* thông tin độ dài packet,
* thông tin offload,
* chaining/scatter-gather context,
* refcnt và trạng thái buffer.

Nếu học DPDK mà không hiểu mbuf, gần như chưa hiểu data path.

## 11.3 Tư duy đúng

Packet trong DPDK không phải “mảng byte tự do”.
Phần lớn pipeline đều vận hành trên **con trỏ `struct rte_mbuf *`**.

---

## 12. Mempool là gì?

## 12.1 Khái niệm

`mempool` là allocator cho các object kích thước cố định, và trong DPDK nó thường được dùng để chứa các **mbuf**.
Thay vì malloc/free packet buffer từng lần, DPDK dùng mempool để:

* cấp phát rất nhanh,
* giảm fragmentation,
* tận dụng cache per-core,
* giảm lock contention.

## 12.2 Vì sao mempool quan trọng?

Hiệu năng của packet path phụ thuộc rất mạnh vào:

* tốc độ cấp phát mbuf,
* tốc độ trả mbuf,
* locality của mempool,
* cache alignment,
* NUMA placement.

## 12.3 Một lỗi cực kỳ phổ biến

Mbuf leak hoặc không free đúng cách sẽ dẫn đến:

* RX dần cạn buffer,
* drop tăng,
* ứng dụng “đang chạy ngon rồi đột nhiên hết packet”,
* hoặc port stop/close bị lỗi logic.

---

## 13. RX burst / TX burst

## 13.1 Vì sao DPDK dùng burst?

Một tư tưởng cực kỳ quan trọng trong DPDK là **xử lý packet theo burst** thay vì từng gói đơn lẻ.
API điển hình là:

```c
uint16_t nb_rx = rte_eth_rx_burst(port_id, queue_id, bufs, BURST_SIZE);
uint16_t nb_tx = rte_eth_tx_burst(port_id, queue_id, bufs, nb_rx);
```

## 13.2 Lợi ích của burst processing

Burst giúp:

* giảm overhead lời gọi hàm trên từng packet,
* tăng locality,
* giúp PMD/vector path tối ưu hơn,
* giảm chi phí thao tác descriptor,
* cải thiện throughput tổng thể.

## 13.3 Trade-off

Burst quá nhỏ:

* latency thấp hơn,
* nhưng throughput có thể kém.

Burst quá lớn:

* throughput tốt hơn,
* nhưng latency per-packet có thể tăng.

Đây là một bài toán tuning thực tế rất quan trọng.

---

## 14. Vòng lặp xử lý packet điển hình trong DPDK

Một data path đơn giản nhất của DPDK thường có dạng:

```c
for (;;) {
    uint16_t nb_rx = rte_eth_rx_burst(port, queue, bufs, BURST_SIZE);
    if (nb_rx == 0)
        continue;

    // xử lý packet
    for (uint16_t i = 0; i < nb_rx; i++) {
        struct rte_mbuf *m = bufs[i];
        // parse / classify / rewrite / forward ...
    }

    uint16_t nb_tx = rte_eth_tx_burst(out_port, out_queue, bufs, nb_rx);

    if (nb_tx < nb_rx) {
        for (uint16_t i = nb_tx; i < nb_rx; i++)
            rte_pktmbuf_free(bufs[i]);
    }
}
```

Đây là “xương sống” của vô số ứng dụng như:

* l2fwd,
* bridge,
* router,
* firewall,
* UPF user-plane fast path,
* packet capture / mirroring / traffic generator.

---

## 15. testpmd và l2fwd

## 15.1 Vì sao hai công cụ này rất quan trọng?

Tài liệu DPDK chính thức có các sample applications như **testpmd** và **l2fwd**, và đây gần như là hai điểm khởi đầu quan trọng nhất cho người học DPDK.
DPDK cũng mô tả sample apps như một “cookbook” để người dùng học và mở rộng tính năng.

## 15.2 testpmd

`testpmd` là công cụ tương tác rất mạnh để:

* thử port configuration,
* test RX/TX,
* thử offload,
* RSS,
* queue config,
* flow rules,
* forwarding modes,
* performance sanity test.

## 15.3 l2fwd

`l2fwd` là ví dụ forward L2 rất cơ bản, cực kỳ hữu ích để đọc code.
Nếu bạn muốn hiểu vòng lặp RX/TX burst của DPDK, `l2fwd` là một trong những ví dụ tốt nhất để bắt đầu.

---

## 16. RSS, multi-queue và scale theo core

## 16.1 Vì sao multi-queue quan trọng?

Một core thường không đủ để xử lý data plane tốc độ rất cao.
Do đó DPDK thường tận dụng:

* nhiều RX queue,
* nhiều TX queue,
* RSS để phân phối flow,
* nhiều worker lcore xử lý song song.

## 16.2 RSS là gì trong ngữ cảnh DPDK?

RSS giúp NIC hash flow và phân tán packet vào nhiều RX queue.
Khi kết hợp với mô hình **1 queue / 1 lcore**, hệ thống có thể scale throughput rất tốt mà vẫn giữ flow affinity.

## 16.3 Tư duy đúng

Scale trong DPDK không chỉ là “bật nhiều thread”, mà là:

* NIC queue layout đúng,
* RSS đúng,
* core affinity đúng,
* mempool đúng NUMA,
* workload partitioning hợp lý.

---

## 17. Offload trong DPDK

## 17.1 Offload là gì?

Nhiều NIC hỗ trợ các offload như:

* checksum offload,
* TSO/segmentation,
* RSS,
* VLAN strip/insert,
* timestamp,
* fast mbuf free,
* flow steering.

## 17.2 Vì sao offload quan trọng?

Dùng đúng offload có thể:

* giảm tải CPU,
* tăng throughput,
* giảm latency,
* đơn giản hóa phần xử lý packet ở user space.

Dùng sai hoặc hiểu sai offload có thể gây:

* packet malformed,
* checksum lỗi,
* throughput giảm,
* hành vi khó debug.

---

## 18. rte_flow là gì?

## 18.1 Khái niệm

`rte_flow` là API của DPDK để mô tả rule phân loại/lọc/steering packet theo cách tương đối thống nhất giữa các NIC hỗ trợ.

## 18.2 Vai trò thực tế

`rte_flow` cho phép:

* match packet theo nhiều trường,
* steering vào queue cụ thể,
* drop/mark/redirect,
* triển khai fast path classification gần phần cứng hơn.

## 18.3 Vì sao quan trọng?

Trong nhiều hệ thống production, hiệu năng không chỉ đến từ RX burst nhanh, mà còn đến từ việc **đẩy classification/filtering xuống NIC** khi có thể.

---

## 19. Multi-process trong DPDK

## 19.1 Primary và secondary process

DPDK hỗ trợ mô hình multi-process, trong đó có khái niệm:

* **primary process**,
* **secondary process**.

Điều này hữu ích cho các bài toán như:

* packet capture/song song debug,
* control plane tách khỏi data plane,
* plugin/tooling cùng truy cập tài nguyên DPDK.

## 19.2 file-prefix quan trọng vì sao?

Khi chạy nhiều DPDK process, `--file-prefix` giúp tách namespace tài nguyên dùng chung như hugepage files/memzones và tránh đụng nhau không chủ ý.

---

## 20. Mẫu chương trình DPDK tối thiểu bằng C

Dưới đây là skeleton tối giản để hình dung cấu trúc một ứng dụng DPDK.
Đây không phải full production code, nhưng đủ để thấy khung tư duy.

```c
#include <stdio.h>
#include <stdint.h>
#include <inttypes.h>
#include <rte_eal.h>
#include <rte_ethdev.h>
#include <rte_mbuf.h>
#include <rte_mempool.h>
#include <rte_launch.h>
#include <rte_lcore.h>

#define NUM_MBUFS 8192
#define MBUF_CACHE_SIZE 256
#define RX_RING_SIZE 1024
#define TX_RING_SIZE 1024
#define BURST_SIZE 32

static struct rte_mempool *pktmbuf_pool;

static int
port_init(uint16_t port, struct rte_mempool *mp)
{
    struct rte_eth_conf port_conf = {0};
    const uint16_t rx_rings = 1, tx_rings = 1;
    int ret;

    ret = rte_eth_dev_configure(port, rx_rings, tx_rings, &port_conf);
    if (ret < 0)
        return ret;

    ret = rte_eth_rx_queue_setup(port, 0, RX_RING_SIZE,
                                 rte_eth_dev_socket_id(port),
                                 NULL, mp);
    if (ret < 0)
        return ret;

    ret = rte_eth_tx_queue_setup(port, 0, TX_RING_SIZE,
                                 rte_eth_dev_socket_id(port),
                                 NULL);
    if (ret < 0)
        return ret;

    ret = rte_eth_dev_start(port);
    if (ret < 0)
        return ret;

    rte_eth_promiscuous_enable(port);
    return 0;
}

static int
lcore_main(void *arg)
{
    uint16_t port = (uint16_t)(uintptr_t)arg;
    struct rte_mbuf *bufs[BURST_SIZE];

    for (;;) {
        uint16_t nb_rx = rte_eth_rx_burst(port, 0, bufs, BURST_SIZE);
        if (nb_rx == 0)
            continue;

        /* Ở đây có thể parse / sửa header / classify / forward */

        uint16_t nb_tx = rte_eth_tx_burst(port, 0, bufs, nb_rx);
        if (nb_tx < nb_rx) {
            for (uint16_t i = nb_tx; i < nb_rx; i++)
                rte_pktmbuf_free(bufs[i]);
        }
    }

    return 0;
}

int
main(int argc, char **argv)
{
    int ret = rte_eal_init(argc, argv);
    if (ret < 0)
        rte_exit(EXIT_FAILURE, "Cannot init EAL\n");

    uint16_t nb_ports = rte_eth_dev_count_avail();
    if (nb_ports == 0)
        rte_exit(EXIT_FAILURE, "No available ports\n");

    pktmbuf_pool = rte_pktmbuf_pool_create("MBUF_POOL",
                                           NUM_MBUFS * nb_ports,
                                           MBUF_CACHE_SIZE,
                                           0,
                                           RTE_MBUF_DEFAULT_BUF_SIZE,
                                           rte_socket_id());
    if (pktmbuf_pool == NULL)
        rte_exit(EXIT_FAILURE, "Cannot create mbuf pool\n");

    for (uint16_t port = 0; port < nb_ports; port++) {
        if (port_init(port, pktmbuf_pool) < 0)
            rte_exit(EXIT_FAILURE, "Cannot init port %u
", port);
    }

    unsigned lcore_id;
    RTE_LCORE_FOREACH_WORKER(lcore_id) {
        rte_eal_remote_launch(lcore_main, (void *)(uintptr_t)0, lcore_id);
    }

    lcore_main((void *)(uintptr_t)0);
    return 0;
}
```

## 20.1 Ý nghĩa của skeleton này

Mẫu trên minh họa những ý cốt lõi nhất của ứng dụng DPDK:

* `rte_eal_init()` để bootstrap môi trường
* `rte_pktmbuf_pool_create()` để tạo mempool chứa mbuf
* `rte_eth_dev_configure()` và queue setup để cấu hình port
* `rte_eth_rx_burst()` và `rte_eth_tx_burst()` cho data path
* mô hình lcore chạy vòng lặp polling vô hạn

## 21. Hiệu năng trong DPDK: những điểm quyết định

## 21.1 Không có một nút thần kỳ

Hiệu năng DPDK thường được quyết định bởi nhiều yếu tố cùng lúc:

* PMD và NIC cụ thể,
* core affinity,
* NUMA placement,
* burst size,
* mempool cache,
* queue count,
* offload,
* cache miss,
* branch miss,
* LLC pressure,
* IOMMU/vfio overhead trong một số cấu hình,
* packet size và traffic pattern.

## 21.2 Những nguyên tắc thực dụng

Một số nguyên tắc rất hay đúng trong thực tế:

* giữ **NIC, queue, mempool, worker** cùng NUMA socket
* dùng **1 RX queue ↔ 1 worker** khi có thể
* tránh chia sẻ lock trên fast path
* ưu tiên **burst processing**
* tránh malloc/free động trên fast path
* tránh log quá nhiều trong data path
* free mbuf đúng cách và đúng nhịp
* đo bằng số liệu thay vì đoán

---

## 22. Các lỗi phổ biến khi mới làm DPDK

### 22.1 Lỗi setup môi trường

* chưa reserve hugepage
* mount hugepage sai
* quên bind NIC sang vfio-pci
* quyền truy cập VFIO không đủ
* EAL args sai

### 22.2 Lỗi data path

* quên free mbuf khi TX không gửi hết
* dùng sai queue id
* cấu hình RX/TX descriptor không hợp lý
* xử lý packet ngay trên core sai NUMA
* mbuf pool quá nhỏ
* cache line contention

### 22.3 Lỗi hiệu năng

* traffic nằm trên NIC socket 1 nhưng worker ở socket 0
* burst size quá nhỏ hoặc quá lớn
* quá nhiều debug log trong loop
* branch rẽ nhánh phức tạp trên hot path
* offload bật/tắt không đúng

### 22.4 Lỗi đa tiến trình

* quên `--file-prefix`
* nhầm primary/secondary
* chia sẻ memzone không đúng

## 23. Góc nhìn debug thực chiến

Khi hệ thống DPDK có triệu chứng như:

* không thấy port,
* RX bằng 0,
* TX không ra packet,
* throughput thấp,
* CPU 100% nhưng packet rate thấp,
* packet drop tăng,
* latency jitter cao,

thì các câu hỏi nên soi lần lượt là:

1. **NIC đã bind đúng vfio-pci chưa?**
2. **Hugepage đã đủ và map đúng chưa?**
3. **Port có start thành công không?**
4. **Link up chưa? queue setup đúng chưa?**
5. **RSS / queue / lcore mapping có hợp lý không?**
6. **Mempool có đủ object không? có leak mbuf không?**
7. **NUMA placement có sai socket không?**
8. **RX burst có thấy packet nhưng bị drop ở app logic không?**
9. **TX burst có trả thiếu và không free phần còn lại không?**
10. **Offload/flow/rte_flow có đang làm thay đổi hành vi không?**
11. **Bottleneck là PMD, app logic, memory hay PCIe/NIC?**

Đây là trình tự debug rất hiệu quả vì nó đi từ nền tảng hệ thống tới data path và cuối cùng mới đi sâu vào tối ưu vi mô.

## 24. DPDK hợp với bài toán nào?

DPDK đặc biệt phù hợp cho:

* L2/L3 forwarding
* vRouter / vSwitch
* firewall / ACL / packet filter
* load balancer
* UPF / user plane telecom
* traffic generator / packet capture tốc độ cao
* DPI / telemetry packet pipeline
* NFV / CNF packet data plane

Ngược lại, nếu bài toán không cần packet rate cao hoặc không chấp nhận mô hình polling ăn CPU, DPDK có thể không phải lựa chọn phù hợp nhất.

## 25. Tóm tắt kỹ thuật ngắn gọn

* **DPDK** là nền tảng packet processing hiệu năng cao trong user space.
* **EAL** bootstrap môi trường runtime.
* **Hugepage** là nền tảng bộ nhớ quan trọng của DPDK.
* **vfio-pci** thường là driver binding thực tế được dùng cho NIC DPDK trên Linux hiện đại.
* **PMD** là driver poll mode chạy trong user space.
* **ethdev** là API abstraction chính cho Ethernet device.
* **mbuf** là packet buffer trung tâm.
* **mempool** là nơi cấp phát object cố định cho mbuf nhanh và hiệu quả.
* **RX/TX burst** là tư tưởng xử lý packet cốt lõi của DPDK.
* **NUMA**, **RSS**, **queue-to-core mapping** và **offload** quyết định hiệu năng thực tế.
* Muốn debug DPDK đúng, phải nhìn toàn chuỗi từ **EAL → hugepage → binding → port/queue → mempool/mbuf → RX/TX burst → NUMA/perf**.

---

## 26. Kết luận

Nếu Linux kernel networking stack là một hệ thống mạng tổng quát, giàu tính năng và tiện lập trình, thì **DPDK** là con đường dành cho khi bạn cần data plane nhanh, kiểm soát sát phần cứng và chấp nhận đánh đổi bằng polling, core pinning và độ phức tạp hệ thống cao hơn.

Muốn hiểu DPDK đúng bản chất, đừng chỉ nhìn nó như thư viện API.
Hãy nhìn nó như một kiến trúc đầy đủ:

**EAL → hugepage → vfio-pci/PMD → ethdev → mempool/mbuf → RX/TX burst → lcore/NUMA → packet pipeline**

Đó mới là cách hiểu DPDK theo đúng tư duy hệ thống, tư duy hiệu năng và tư duy implementer C/Linux.

## 28. Cách nhìn rất ngắn để nhớ

Nếu cần nhớ cực ngắn theo kiểu phỏng vấn hoặc đọc code:

* **EAL**: khởi động môi trường DPDK
* **Hugepage**: bộ nhớ lớn, nhanh, thân thiện DMA
* **vfio-pci**: bind NIC cho user-space driver
* **PMD**: poll NIC từ user space
* **ethdev**: API thao tác port/queue
* **mbuf**: packet buffer
* **mempool**: pool cấp phát mbuf
* **RX/TX burst**: vòng lặp xử lý packet cốt lõi
* **NUMA**: locality quyết định hiệu năng

Đó là lõi của toàn bộ câu chuyện **DPDK**.
